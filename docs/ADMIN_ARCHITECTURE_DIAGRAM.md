### Migration Guide

```bash

# or
pnpm add jwt-decode
```

#### Step 2: Update User Roles in Supabase

**Option A: Via Supabase Dashboard**
1. Authentication â†’ Users â†’ Select user
2. Raw User Meta Data â†’ Add: `{"role": "admin"}`
3. Click Save

**Option B: Via SQL**
```sql
-- Update auth metadata (for JWT)
UPDATE auth.users
SET raw_user_meta_data = raw_user_meta_data || '{"role":"admin"}'::jsonb
WHERE email = 'admin@example.com';

-- Update profiles table (for DB queries)
UPDATE profiles
SET role = 'admin'
WHERE email = 'admin@example.com';
```

### Production Considerations

1. **JWT Signature Verification**
   - Current implementation uses `jwt-decode` (no verification)
   - Production MUST verify signatures using Supabase public key
   - Use `@supabase/jwt-verify` or similar

2. **Token Expiry**
   - Consider shorter token lifetime (default: 1 hour)
   - Configure in Supabase Dashboard â†’ Settings â†’ Auth

3. **Role Change Notifications**
   - Implement real-time updates via Supabase Realtime
   - Or force sign-out after role changes

4. **Audit Logging**
   - Log all role checks with IP, timestamp
   - Alert on repeated authorization failures

5. **Rate Limiting**
   - Add rate limiting to auth endpoints
   - Prevent brute force role escalation attempts

**Bug Fix Summary:**
This update centralizes role handling, adds comprehensive debugging, and implements a token-first authorization approach with database fallback. The key insight is that JWT tokens contain stale role information after database updates, requiring explicit token refresh (re-login) to resolve the "can view but cannot act" issue.
